## 1. 平台概述

**仓储效率治理与报价平台**（wh-op-platform）是专为第三方物流（3PL）仓储运营场景设计的数字化管理工具。平台以 MySQL 业务数据为输入，通过 Python 数据分析服务进行预计算，再由 Java 后端聚合展示，最终通过 Vue3 前端提供交互界面。

### 核心价值

| 能力 | 说明 |
|------|------|
| **费用透明化** | 将劳务工时与报价单价量化为可追溯的月度费用基线，消除"成本黑盒" |
| **效率量化** | 人效、件单比、日均单量等 KPI 横向对比多仓，定位低效节点 |
| **智能归因** | 通过 Pearson 相关性分析，自动识别影响工时投入的核心驱动因素 |
| **报价测算** | 五步法成本估算模型，支持调参预测不同业务场景下的月度费用 |
| **报告自动化** | 一键生成结构化 Markdown 分析报告，可下载为 HTML 存档 |

### 当前接入仓库

| 仓库编码 | 仓库名称 |
|----------|----------|
| `12000004` | 天津武清佩森A仓 |
| `32050005` | 常熟高新正创B仓 |

### 2.2 服务职责划分

| 服务 | 技术栈 | 职责 |
|------|--------|------|
| **Frontend** | Vue 3, TypeScript, Vite, Ant Design Vue 4, ECharts | UI 展示、用户交互、数据可视化 |
| **Backend** | Spring Boot 3.2, Java 17, MyBatis-Plus | REST API 聚合、业务逻辑计算、报告生成 |
| **Analytics** | Python 3.11, FastAPI, APScheduler, pymysql, SQLite | 大数据量预计算、Pearson 相关性分析、缓存服务 |
| **MySQL** | MySQL 8.x | 业务原始数据存储 |
| **SQLite** | SQLite 3 (WAL 模式) | Analytics 预计算结果缓存 |

### 2.3 数据流向

```
MySQL (源数据)
  ↓ Analytics 每 6h 预计算
SQLite (daily_metrics / baseline_results / impact_results)
  ↓ FastAPI 读取
  ↓ Spring Boot 调用 FastAPI / 直接查 MySQL
  ↓ Vue 前端调用 Spring Boot
  ↓ 页面展示
```

**容错设计：** Analytics 服务停机时，Backend 的 `AnalyticsClient` 会捕获所有异常并返回空集合，其余模块（数据看板、费用基线、成本估算）仍可正常运行。

## 3. 数据源与数据库设计

### 3.1 MySQL 业务表

#### 3.1.1 出库单表（Outbound Orders）

记录仓库每笔出库业务单据，是出库量统计的核心数据源。

| 关键字段 | 类型 | 用途 |
|----------|------|------|
| 库房编码 | VARCHAR | 仓库标识（主连接键） |
| 库房名称 | VARCHAR | 仓库名称 |
| 创建时间 | DATETIME | 按月聚合的时间基准 |
| 物料总数量 | DECIMAL | 该单的出库件数 |
| 状态 | VARCHAR | `"撤单"` 状态的单据需排除 |

**关键查询：**
```sql
SELECT DATE_FORMAT(创建时间, '%Y-%m') AS month,
       COUNT(*) AS total_orders,
       SUM(物料总数量) AS total_items
FROM 出库单表
WHERE 库房编码 = ? AND status != '撤单'
GROUP BY month
```

---

#### 3.1.2 出勤统计表（Attendance Statistics）

记录仓库每日员工出勤与工时数据，是费用计算和人力分析的核心数据源。

| 关键字段 | 类型 | 用途 |
|----------|------|------|
| 库房 | VARCHAR | 仓库名称（与出库单通过 fuzzy match 联接） |
| 员工编码 | VARCHAR | 唯一员工标识，用于去重计人数 |
| **员工类型** | VARCHAR | `"自有人员"` / `"长期劳务"` / `"临时劳务"` |
| 考勤日期 | DATE | 每日考勤记录 |
| **工作时长** | VARCHAR | **单位：分钟**，存储为字符串，需转型后求和 |

**员工类型分类规则：**

| 类别 | 员工类型值 | 说明 |
|------|-----------|------|
| 固定人员 | `自有人员`, `长期劳务` | 月度稳定用工 |
| 临时人员 | `临时劳务` | 弹性临时用工 |

**关键查询：**
```sql
SELECT DATE_FORMAT(考勤日期, '%Y-%m') AS month,
       SUM(CAST(工作时长 AS DECIMAL)) / 60 AS total_work_hours,
       COUNT(DISTINCT 员工编码) AS avg_headcount,
       COUNT(DISTINCT 考勤日期) AS work_days,
       SUM(CASE WHEN 员工类型 IN ('自有人员','长期劳务') THEN 1 ELSE 0 END) AS fixed_count,
       SUM(CASE WHEN 员工类型 = '临时劳务' THEN 1 ELSE 0 END) AS temp_count
FROM 出勤统计表
WHERE 库房 LIKE CONCAT('%', ?, '%')
GROUP BY month
```

---

#### 3.1.3 报价信息表（Quotation Info）

存储与各仓库签订的劳务报价合同，为费用计算提供单价基准。

| 关键字段 | 类型 | 用途 |
|----------|------|------|
| 库房名称 | VARCHAR | 仓库名称（用于匹配） |
| **结费类型** | VARCHAR | `"固定劳务"` / `"临时劳务"` / `"叉车劳务"` |
| **供应商结算单价** | DECIMAL | 劳务单价（元/小时） |
| **报价状态** | VARCHAR | 仅 `"正常"` 状态参与计算 |
| 税率 | DECIMAL | 适用税率（默认 6%） |

**关键查询（加权平均单价）：**
```sql
SELECT AVG(供应商结算单价) AS weighted_unit_price
FROM 报价信息表
WHERE 库房名称 LIKE CONCAT('%', ?, '%')
  AND 报价状态 = '正常'
```

> 若无匹配报价数据，回退默认值 `20.00 元/h`。

#### 3.1.4 入库单表（Inbound Orders）

#### 3.1.5 上架单表（Shelving Orders）

记录入库后的商品上架操作，关联入库单。

| 关键字段 | 类型 | 用途 |
|----------|------|------|
| 库房编码 | VARCHAR | 仓库标识 |
| 创建时间 | DATETIME | 时间基准 |
| 上架单总数量 | DECIMAL | 上架件数 |

---

#### 3.1.6 退货信息表（Return Info）

记录退货入库单据，通过入库单号关联入库单。

## 4. 核心指标与通用计算公式

以下公式为整个平台所有模块的计算基础。

### 4.1 月度劳务总费用

$$\text{月度总费用} = \text{月总工时(h)} \times \text{加权平均单价(元/h)} \times (1 + \text{税率})$$

其中税率默认为 6%（即 ×1.06）。

### 4.2 加权平均劳务单价

$$\text{加权单价} = \frac{\sum \text{供应商结算单价}}{\text{有效报价条数}}$$

从报价信息表中取 `报价状态='正常'` 的记录，按仓库聚合后取算术平均值。

### 4.3 总工时计算

出勤统计表中工作时长字段单位为**分钟**：

$$\text{总工时(h)} = \frac{\sum \text{工作时长(分钟)}}{60}$$

### 4.4 人效（劳动效率）

$$\text{人效} = \frac{\text{月出库单量}}{\text{平均在岗人数} \times \text{工作天数}} \quad \text{(件/人·日)}$$

### 4.5 固临比（固定/临时人员比例）

$$\text{固临比} = \frac{\text{固定人员数（自有+长期劳务）}}{\text{临时劳务人数}}$$

### 4.6 件单比

$$\text{件单比} = \frac{\text{月总出库件数}}{\text{月总出库单量}}$$

### 4.7 仓库类型分类

系统根据三个维度自动为每个仓库打标签：

| 维度 | 判断标准 | 标签 |
|------|---------|------|
| **规模**（月出库单量） | > 200,000 | 大型仓 |
| | > 100,000 | 中大型仓 |
| | > 50,000 | 中型仓 |
| | 其余 | 小型仓 |
| **件单比** | < 3 | 低件单比 |
| | 3 ~ 6 | 中件单比 |
| | > 6 | 高件单比 |
| **劳务结构**（固临比） | > 1.0 | 固临均衡型 |
| | 0.5 ~ 1.0 | 混合型 |
| | < 0.5 | 临时主导型 |

最终标签拼接为形如 `"中型仓 / 中件单比 / 混合型"` 的字符串。

## 5. 模块详解：数据看板

### 5.1 功能概述

数据看板是平台首页，提供选定仓库在选定月度的运营全景视图，分为 KPI 卡片区、折线趋势图、费用结构图和工作量分布图四个区域。

### 5.2 页面交互

- 顶部导航栏的**仓库选择器**全局控制所有页面展示的仓库
- **月份选择器**从可用月份列表中选择（系统自动默认最新月）
- 月份或仓库变更时，所有图表联动刷新
- 所有图表支持 ECharts 骨架屏加载动效

### 5.3 KPI 卡片（Overview）

**数据接口：** `GET /api/dashboard/overview?warehouseCode=&month=`

| 卡片 | 字段 | 计算说明 |
|------|------|---------|
| 总单量 | totalOrders | 选月内非撤单出库单数量 |
| 总工时 | totalWorkHours | `SUM(工作时长) / 60` |
| 月度费用 | monthlyFee | `totalWorkHours × weightedUnitPrice × 1.06` |
| 人效 | laborEfficiency | `totalOrders / (avgHeadcount × workDays)` |
| 日均费用 | dailyAvgFee | `monthlyFee / workDays` |
| 出勤天数 | workDays | `COUNT(DISTINCT 考勤日期)` |
| 件单比 | itemsPerOrder | `totalItems / totalOrders` |
| 单均成本 | avgCostPerOrder | `monthlyFee / totalOrders` |

### 5.4 折线趋势图（日粒度出库量）

**数据接口：** `GET /api/dashboard/trend?type=outbound_orders`

展示所选月份内每日出库单量的折线变化，帮助识别业务峰谷规律（如周末低谷、促销大促高峰）。

### 5.5 费用结构堆叠柱图（月度费用拆解）

**数据接口：** `GET /api/dashboard/trend?type=monthly_fee_breakdown`

按员工类型（固定劳务 / 临时劳务 / 叉车劳务）拆解月度费用构成，以堆叠柱形图展示，直观反映各类劳务的费用占比及变化趋势。

### 5.6 工作量分布饼图

**数据接口：** `GET /api/dashboard/trend?type=workload_distribution`

根据工作量统计操作明细表的**操作大类**字段，将月度操作量按类型聚合为环形图，反映各作业类型的工时投入占比。

> **分类归一化规则：** 原始 `操作大类` 字段包含 7 个值（出库、手持出库、入库、手持入库、在库、手持在库、退货），约 37% 的记录该字段为 NULL。系统采用统一归一化逻辑：
> - 手持出库 → 出库、手持入库 → 入库、手持在库 → 在库
> - NULL 记录根据 `单据类型` 字段自动推断：含"入库"且不含"退货" → 入库；含"退货" → 退货；含"转储/加工/库存/拆包" → 在库；其余 → 出库
> - 最终归一化为 4 个标准类别：**出库、入库、在库、退货**

## 6. 模块详解：费用基线

### 6.1 功能概述

费用基线模块用于查看指定月份区间内的历史成本基线、每日费用明细与双仓趋势对比，是制定报价标准、校验运营波动和识别仓库差异的核心分析入口。

### 6.2 月度基线表格

**数据接口：** `GET /api/baseline/monthly?warehouseCode=&startMonth=&endMonth=&page=&size=`

页面顶部提供 3 个查询条件：
- **月份区间**：开始月份 + 结束月份，格式 `YYYY-MM`
- **仓库筛选**：可选，不选时表示所有仓库
- **查询按钮**：触发区间级刷新

**初始化行为：** 页面加载时自动调用 `GET /api/baseline/latestMonth` 获取最新数据月，并默认展示“当前所有仓库最近 12 个月”的月度基线数据，即：
- `endMonth = latestMonth`
- `startMonth = latestMonth - 11 months`
- 列表按 `month DESC` 排序展示

表格分页展示（默认每页 20 条），包含以下所有指标：

| 列名 | 字段 | 单位 | 计算公式 |
|------|------|------|---------|
| 仓库 | warehouseName | - | - |
| 年份 | year | - | - |
| 月份 | month | - | - |
| 日均费用 | dailyAvgFee | 元 | `totalFee / workDays` |
| 总费用 | totalFee | 元 | `totalWorkHours × weightedUnitPrice × 1.06` |
| 总单量 | totalOrders | 单 | `COUNT(*)` 出库单 |
| 总件数 | totalItems | 件 | `SUM(物料总数量)` |
| 件单比 | itemsPerOrder | - | `totalItems / totalOrders` |
| 日均单量 | dailyAvgOrders | 单/日 | `totalOrders / workDays` |
| 单均成本 | costPerOrder | 元/单 | `totalFee / totalOrders` |
| 件均成本 | costPerItem | 元/件 | `totalFee / totalItems` |
| 平均人数 | avgHeadcount | 人 | `COUNT(DISTINCT 员工编码)` |
| 人效 | laborEfficiency | 件/人时 | `totalOrders / (avgHeadcount × workDays)` |
| 固临比 | fixedTempRatio | - | `fixedCount / tempCount` |
| 总工时 | totalWorkHours | h | `SUM(工作时长) / 60` |
| 加权单价 | weightedUnitPrice | 元/h | `AVG(供应商结算单价)` |

该表格用于回答两个核心问题：
- 当前区间内，不同仓库在不同月份的成本水平是否稳定
- 成本差异究竟来自工时变化、单价变化，还是业务单量结构变化

### 6.3 每日操作费用明细表

**数据接口：** `GET /api/baseline/dailyDetail?warehouseCode=&startMonth=&endMonth=`

在月度基线总览下方，页面展示当前月份区间内按“天”展开的费用分析明细表，帮助用户进一步定位月度费用异常是由哪几天驱动。

表格字段如下：
- 日期
- 仓库
- 出库单量
- 出库件数
- 件单比
- 出勤人数
- 工时(h)
- 当日费用(¥)

其中关键计算关系为：
- `itemOrderRatio = obItems / obOrders`
- `dailyFee = workHours × weightedUnitPrice × 1.06`

该明细表替代了原先的“加权平均劳务单价对比柱图”。原因是单价本身更适合作为成本组成字段，而不是单独的视觉主图；日级明细更适合用于解释区间内的实际波动来源。

### 6.4 双仓月度对比

**数据接口：** `GET /api/baseline/compare?codes={A},{B}&startMonth=&endMonth=`

用户选择两个仓库后，系统返回当前月份区间内“每仓每月”一条对比记录，并以月份作为 X 轴，绘制双仓月度趋势对比图。

**可切换指标：**

**图表逻辑：**
- X 轴：月份区间内的每个月，格式 `YYYY-MM`

- Y 轴：当前选中指标的值

这意味着双仓对比不再是“某一个月的 5 个指标横向对比”，而是“某一指标在多个自然月上的趋势对比”。

### 6.5 对比指标计算口径

为避免旧版页面中出现总费用、总单量、平均人数展示错误，双仓对比接口现已补齐并统一以下字段：

### 6.6 操作大类费用明细

**数据接口：** `GET /api/baseline/operation-fee-detail?warehouseCode=&startMonth=&endMonth=&dimension=month|day`

该接口将月度/日度总操作费用按**操作大类**进行分摊，揭示不同作业类型对总费用的贡献占比。

**计算逻辑：**

1. **计算总费用**
   - 月度维度：`总费用 = SUM(出勤工时) ÷ 60 × 加权单价 × 1.06`（按月、按仓库聚合出勤统计表）
   - 日度维度：复用 `dailyDetail` 接口的每日费用

2. **统计操作量**
   - 数据源：`工作量统计操作明细表`，按 `工厂编码` 过滤
   - 按时间维度（月/日）和**归一化操作大类**分组，`COUNT(*)` 计算每类操作量

3. **操作大类归一化（CASE 表达式）**

| 原始 `操作大类` 值 | 归一化结果 | 说明 |
|-----------|--------|------|
| 出库、手持出库 | 出库 | 合并手持终端操作 |
| 入库、手持入库 | 入库 | 合并手持终端操作 |
| 在库、手持在库 | 在库 | 合并手持终端操作 |
| 退货 | 退货 | 保持不变 |
| NULL | 根据 `单据类型` 推断 | 含"入库"(非退货) → 入库；含"退货" → 退货；含"转储/加工/库存/拆包" → 在库；其余 → 出库 |

4. **费用分摊**
   - `费用占比 = 该类操作量 / 总操作量`
   - `估算费用 = 总费用 × 费用占比`

**返回字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| dimension | string | `month` 或 `day` |
| date | string | 日期（仅日度维度） |
| monthLabel | string | `YYYY-MM` |
| warehouseCode | string | 仓库编码 |
| warehouseName | string | 仓库名称 |
| operationCategory | string | 归一化操作大类（出库/入库/在库/退货） |
| operationCount | long | 操作量 |
| feeRatio | decimal | 费用占比 |
| estimatedFee | decimal | 估算费用（元） |

## 7. 模块详解：影响因素分析

### 7.1 功能概述

影响因素分析模块利用统计学中的 Pearson 相关性分析，量化各业务指标对**总工时**的影响程度，帮助管理者识别真正驱动人力投入的核心变量，从而制定针对性的效率优化策略。

### 7.2 分析方法：Pearson 相关系数

对于某个候选因素 $x$（如出库单量）和目标变量 $y$（总工时），以日粒度历史数据作为样本，计算 Pearson 相关系数：

$$r = \frac{\sum_{i=1}^{n}(x_i - \bar{x})(y_i - \bar{y})}{\sqrt{\sum_{i=1}^{n}(x_i-\bar{x})^2} \cdot \sqrt{\sum_{i=1}^{n}(y_i-\bar{y})^2}}$$

- $r \in [-1, 1]$，取绝对值 $|r|$ 衡量影响强度
- $r > 0$ 表示正相关（该指标上升时工时上升）
- $r < 0$ 表示负相关（该指标上升时工时下降）
- 最小有效样本数：**5 个**

### 7.3 分析的 10 个候选因素

| 因素名称 | 对应字段 | 说明 |
|----------|---------|------|
| 出库单量 | ob_orders | 当日出库订单数 |
| 出库件数 | ob_items | 当日出库件数总量 |
| 件单比 | item_order_ratio | 出库件数/出库单量 |
| 入库单量 | ib_orders | 当日入库单数 |
| 退货量 | return_orders | 当日退货入库量 |
| 出勤人数 | headcount | 当日出勤总人数 |
| 固定劳务人数 | fixed_count | 当日固定人员数 |
| 临时劳务人数 | temp_count | 当日临时人员数 |
| 固临比 | fixed_temp_ratio | 固定/临时人员比例 |
| 上架量 | shelf_orders | 当日上架操作单量 |

### 7.4 三梯队分类体系

| 梯队 | 阈值条件 | 标签 | 颜色 |
|------|---------|------|------|
| 第一梯队 | $|r| \geq 0.7$ | 核心驱动 | 红色 |
| 第二梯队 | $0.4 \leq |r| < 0.7$ | 重要影响 | 橙色 |
| 第三梯队 | $|r| < 0.4$ | 结构影响 | 灰色 |

### 7.5 页面功能详解

#### 因素排名横向条形图

以 $|r|$ 排序，按因素绑定梯队颜色，直观呈现各因素对工时的影响强弱。

#### 因素详情表格

| 列 | 说明 |
|----|------|
| 排名 | 按 $|r|$ 降序排列 |
| 因素名称 | 业务可读名称 |
| 相关系数 | Pearson $r$ 值（保留两位小数） |
| 梯队 | 彩色标签（核心驱动/重要影响/结构影响） |
| 应用建议 | 根据梯队和相关方向提供的优化提示 |

#### 10×10 相关性热力图

展示所有 10 个候选因素两两之间的 Pearson 相关系数矩阵，色阶从蓝（$r=-1$）→白（$r=0$）→红（$r=+1$），帮助识别因素间的共线性关系。

#### 单因素散点图

用户从下拉框选择具体因素后，加载该因素值（X 轴）vs 当日总工时（Y 轴）的散点图，每个点对应一条日粒度记录，鼠标悬停显示具体日期，直观验证相关性。

**数据接口：** `GET /api/impact/scatter?warehouseCode=&factor=`

#### 双仓因素对比

选择两个仓库，并排显示各自的因素排名横向条形图，用于横向比较不同仓库的驱动因素差异。

## 8. 模块详解：成本估算

### 8.1 功能概述

成本估算模块提供**五步法报价测算模型**，允许运营/商务人员基于预期业务量调整参数，快速得出月度费用预估值，用于合同谈判与内部预算制定。

### 8.2 五步测算法

#### 输入参数

| 参数 | 默认值来源 | 说明 |
|------|-----------|------|
| 日均单量 | 历史数据 | 预期每日处理出库单数 |
| 件单比 | 历史数据 | 预期每单平均件数 |
| **月工作天数** | **26 天**（固定） | 月度有效工作日 |
| 人效 | 历史数据 | 单位：件/人时 |
| 每日工时 | 8 h | 人均每日工作时长 |
| 税率 | 6% | 劳务税率 |
| 固定劳务单价 | 历史报价 | 元/h |
| 临时劳务单价 | 历史报价 | 元/h |
| 叉车劳务单价 | 历史报价 | 元/h |
| 固定劳务占比 α | 历史数据 | 0~1 |
| 临时劳务占比 β | `1 - α - γ` | 自动计算 |
| 叉车劳务占比 γ | 历史数据 | 0~1 |
| 件单比修正开关 | `false` | 是否启用件单比修正 |
| 基线件单比 | 历史数据 | 仅在修正开启时使用 |

> **历史数据**默认值：页面加载时通过 `GET /api/estimate/defaults/{warehouseCode}` 从数据库中获取。`月工作天数` 字段固定初始化为 26，不被历史数据覆盖。

---

#### 步骤 1：计算基础人力需求

$$N_{base} = \left\lceil \frac{\text{日均单量}}{\text{人效}} \right\rceil$$

> 取上取整（`Math.ceil`）确保人力不低于需求。

---

#### 步骤 2：件单比修正系数（可选）

$$K = \begin{cases} 1.0 & \text{若未开启修正} \\ 1 + 0.05 \times \max(0,\ \text{件单比} - \text{基线件单比}) & \text{若开启修正} \end{cases}$$

**含义：** 当实际件单比超过基线时，每超出 1 的件单比使人力系数额外增加 5%，以反映更高的拣货/打包复杂度。

---

#### 步骤 3：估算实际人数

$$N_{est} = \left\lceil N_{base} \times K \right\rceil$$

---

#### 步骤 4：月度总工时

$$H = N_{est} \times \text{每日工时} \times \text{月工作天数}$$

$$H = N_{est} \times 8 \times 26$$

---

#### 步骤 5：加权平均单价

$$P = \alpha \times P_{固定} + \beta \times P_{临时} + \gamma \times P_{叉车}$$

其中 $\alpha + \beta + \gamma = 1$。

---

#### 最终费用计算

$$\text{月度费用} = H \times P \times (1 + \text{税率})$$

---

#### 输出指标

| 指标 | 计算公式 |
|------|---------|
| 预估人数 | $N_{est}$ |
| 件单比修正系数 | $K$ |
| 预估总工时 | $H$ (h) |
| 加权平均单价 | $P$ (元/h) |
| 月度费用 | $F$ (元) |
| 日均费用 | $F / \text{工作天数}$ |
| 估算总单量 | $\text{日均单量} \times \text{工作天数}$ |
| 单均成本 | $F / \text{估算总单量}$ |
| 估算总件数 | $\text{日均单量} \times \text{件单比} \times \text{工作天数}$ |
| 件均成本 | $F / \text{估算总件数}$ |

### 8.3 基线对比

右侧结果面板还展示当前仓库最新历史月的**基线日均费用**，并计算估算值与基线的偏差百分比：

$$\text{偏差\%} = \frac{\text{估算日均费用} - \text{基线日均费用}}{\text{基线日均费用}} \times 100\%$$

偏差为正（估算高于历史）时显示红色警示。

### 8.4 估算 vs 历史对比图

页面下方以分组柱形图对比估算值与历史实际值（最近一个月基线）的四项指标：月度费用、日均费用、单均成本、件均成本。

## 10. 数据预计算流水线（Analytics 服务）

### 10.1 触发机制

| 触发场景 | 时机 |
|----------|------|
| 服务启动时 | 应用初始化完成后立即执行一次 |
| 定时调度 | 每隔 **6 小时** 由 APScheduler 自动触发 |

### 10.2 流水线三阶段

#### 阶段一：每日指标预计算 (`compute_daily_metrics`)

```
1. 查询 MySQL 出库单表  → 按日聚合 ob_orders, ob_items
2. 查询 MySQL 入库单表  → 按日聚合 ib_orders, ib_items
3. 查询 MySQL 退货信息表 → 按日聚合 return_orders
4. 查询 MySQL 上架单表  → 按日聚合 shelf_orders, shelf_items
5. 查询 MySQL 出勤统计表 → 按日聚合 headcount, total_work_hours,
                           fixed_count, temp_count, own_count
6. 以 (date, warehouse_code) 为键，Merge 所有数据
7. 计算派生字段：
   item_order_ratio = ob_items / ob_orders
   fixed_temp_ratio = fixed_count / temp_count
8. UPSERT → SQLite daily_metrics
```

#### 阶段二：月度基线预计算 (`compute_baseline`)

```
1. 读 SQLite daily_metrics，GROUP BY month, warehouse_code
2. 汇总：total_orders, total_items, total_work_hours
3. working_days = COUNT(DISTINCT days WHERE ob_orders > 0)
4. 查询 MySQL 报价信息表获取 avg_unit_price
5. estimated_fee = total_work_hours × avg_unit_price × 1.06
6. cost_per_order = estimated_fee / total_orders
7. cost_per_item = estimated_fee / total_items
8. UPSERT → SQLite baseline_results
```

#### 阶段三：影响因素计算 (`compute_all_impacts`)

```
对每个已知仓库执行：
1. 读 SQLite daily_metrics WHERE total_work_hours > 0 AND ob_orders > 0
2. 若样本数 < 5，跳过（数据不足）
3. 对 10 个候选因素逐一计算 Pearson 相关系数 r(factor, total_work_hours)
4. 按 |r| 降序排列，赋予 rank 1~10
5. 构建 10×10 全因素相关矩阵（存为 JSON）
6. 生成梯队标签和应用建议文本
7. UPSERT → SQLite impact_results
```
